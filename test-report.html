Why this matters: The default CharSet for DllImport is CharSet.None (equivalent to CharSet.Ansi on most platforms). Calling a Unicode Win32 API without CharSet.Unicode silently marshals strings as ANSI, corrupting non-ASCII characters and potentially causing security-relevant data truncation.

Fix: Add CharSet = CharSet.Unicode to the DllImport attribute, or better yet use the W-suffix Unicode variant of the Win32 function explicitly (e.g., CreateFileW instead of CreateFile).
